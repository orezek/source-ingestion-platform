# Configuration Model v2.2

Status: Draft  
Owner: Platform  
Date: 2026-03-09  
Target Release: v2.2

## 1) Purpose

Define a focused v2.2 scope for pipeline configuration UX and contract behavior, based on decisions captured in section 3.5 of `v2-1-delivered-and-v3-scope.md`.

## 2) Scope

This spec covers only:

1. Pipeline creation/editing model.
2. Search space creation/editing model.
3. Runtime profile creation/editing model.
4. Mode-driven field gating and validation.
5. Structured output interaction with `allowInactiveMarking`.
6. Operator sink vs system sink ownership.
7. Sink-keyed shared worker Mongo connection strategy.
8. Downloadable JSON artifact ownership, routing, and Run Detail UI access.
9. Pipeline deletion preconditions and cascade purge behavior.

Out of scope for this spec:

1. Non-artifact GCP infrastructure ownership changes.
2. Pub/Sub orchestration changes.
3. Cancel/drain flow changes.

## 3) Product Decisions

### 3.1 Single Source of Truth for Ingestion On/Off

1. `pipeline.mode` is the only ingestion enablement switch.
2. Runtime profile must not contain `enableIngestion`.
3. Valid modes remain:
   1. `crawl_only`
   2. `crawl_and_ingest`

### 3.2 IDs Are System-Owned

1. `pipelineId`, `searchSpaceId`, and `runtimeProfileId` are system-generated.
2. Operator must not enter or edit these IDs.
3. Pipeline creation form must not display `searchSpaceId` or `runtimeProfileId`.

### 3.3 Post-Create Editability

After creation, Pipeline Detail must expose editable sections for all relevant domains:

1. Pipeline: `name`, `mode`.
2. Search Space: `name`, `maxItems`, `allowInactiveMarking` (output-gated), `description`.
3. Runtime Profile: `name`, `crawler.maxConcurrency`, `crawler.maxRequestsPerMinute`, `ingestion.concurrency` (mode-gated).
4. Operator Sink (MongoDB): `mongodbUri` (write-only in UI and APIs), `dbName`.

### 3.4 Mode-Gated Runtime Behavior

1. When `pipeline.mode = crawl_only`:
   1. `ingestion.concurrency` is visible but not editable.
   2. Ingestion settings are not applied at execution time.
2. When `pipeline.mode = crawl_and_ingest`:
   1. `ingestion.concurrency` is editable.
   2. Ingestion settings are applied at execution time.

### 3.5 Output-Gated Inactive Marking

1. `allowInactiveMarking` is editable only when structured output includes `mongodb`.
2. If output is `downloadable_json` only, `allowInactiveMarking` is disabled and must resolve to `false`.

### 3.6 Logging Verbosity Ownership

1. Runtime profile must not expose `debugLog` or `debugLogsEnabled` to operators in v2.2.
2. Logging verbosity is controlled only by per-service environment configuration.
3. Services continue using their own `LOG_LEVEL` env variable (`control-service-v2`, `crawler-worker-v2`, `ingestion-worker-v2`).

### 3.7 Sink Ownership and Execution Routing

1. System sink is owned by control-service only and configured by one control-service MongoDB URI env.
2. System sink database and collection names are system-generated by control-service.
3. System sink stores control-plane records (`pipelines`, `runs`, and runtime events projections).
4. Operator sink is configured on pipeline as MongoDB `mongodbUri` + `dbName`.
5. For v2.2 MVP, `mongodbUri` remains a plain string (no encryption/secret vault abstraction yet).
6. Operator sink collections are system-defined and not operator-editable:
   1. `crawl_run_summaries`
   2. `ingestion_run_summaries`
   3. `normalized_jobs`
7. Workers receive operator sink routing in `StartRun` payload.
8. Worker bootstrap `MONGODB_URI` env is removed in v2.2.
9. Worker Mongo clients are shared per sink key (`mongodbUri` + `dbName`), typically one active client per pipeline sink.
10. Run lifecycle remains explicit via client acquire/release:
    1. acquire sink client on run start,
    2. release sink client on finalize/cancel/error/expire.
11. Sink client manager uses reference counting and idle TTL close when ref-count reaches zero.
12. Pool and churn guardrails are required:
    1. explicit `maxPoolSize`,
    2. explicit `maxConnecting`,
    3. explicit `waitQueueTimeoutMS`,
    4. max distinct active sink clients per worker process.
13. If sink configuration changes, new runs use a new sink key and previous sink client is drained and closed by idle lifecycle.

### 3.8 Artifact Storage (HTML + JSON)

1. Artifact storage is system-owned, same ownership model as control-plane system Mongo sink.
2. Default artifact delivery backend is GCS.
3. GCS bucket and prefix are configured only in control-service env.
4. Workers must not own artifact bucket/prefix in worker `.env`.
5. Pipeline ID is immutable and must namespace all pipeline artifact resources.
6. Artifact object paths are run-scoped and pipeline-scoped:
   1. HTML dumps: `pipelines/{pipelineId}/runs/{runId}/artifacts/html/...`
   2. Downloadable JSON: `pipelines/{pipelineId}/runs/{runId}/outputs/downloadable-json/...`
7. Control-service resolves run-scoped artifact delivery and passes:
   1. crawler HTML sink via crawler `StartRun.artifactSink`,
   2. downloadable JSON sink via ingestion `StartRun.outputSinks[].delivery`.
8. Control-service stores per-run JSON artifact metadata index for UI listing and download operations.
9. Operator accesses JSON artifacts only through Control Center Run Detail UI.

### 3.9 Immutable and Restricted Pipeline Fields

1. System-managed fields are immutable:
   1. `pipelineId`
   2. `searchSpaceId`
   3. `runtimeProfileId`
   4. structured output destination IDs
   5. `version`, `createdAt`, `updatedAt`, and lifecycle timestamps
   6. run history, projections, counters, and run summaries
2. `source` is editable on create, but not editable after pipeline creation.
3. `mode` is editable only when there is no active run for the pipeline.
4. Operator sink `mongodbUri` and `dbName` are editable only when there is no active run for the pipeline.
5. `mode` and sink changes apply to subsequent runs, never to an already active run.
6. When run status is active (`queued`, `running`, or `stopping`), pipeline configuration updates are blocked.

### 3.10 Pipeline Deletion and Cascade Purge

1. Pipeline deletion is destructive and operator-owned.
2. Pipeline can be deleted only when there are no active runs and no unsettled runtime events for that pipeline.
3. Operator must cancel active runs first and wait for cancel-and-drain completion before deletion.
4. Deletion must fail fast when preconditions are not met.
5. Successful deletion cascades all related data and metadata:
   1. control-plane pipeline metadata,
   2. all run records and run event projections,
   3. all persisted logs held in system storage for pipeline runs,
   4. operator sink database data for the pipeline,
   5. all GCS artifacts under pipeline prefix (HTML dumps and downloadable JSON).
6. No orphan Pub/Sub events are expected because deletion is blocked until events are settled.
7. Deletion execution model is asynchronous job-based purge (not synchronous request-thread purge).
8. Delete job states are:
   1. `deleting`
   2. `deleted`
   3. `delete_failed`
9. Delete job retry policy uses bounded retries with exponential backoff:
   1. default max attempts: `5`,
   2. default initial backoff: `1000ms`,
   3. default max backoff: `30000ms`.
10. v2.2 does not support force-delete bypass of active/unsettled preconditions.

## 4) v2.2 UI Requirements

### 4.1 Pipeline Create

1. Show only operator-facing fields.
2. Do not render inputs for `searchSpaceId` or `runtimeProfileId`.
3. Preserve current pipeline fields:
   1. `name`
   2. `source`
   3. `mode`
4. When MongoDB destination is selected, show sink fields:
   1. `mongodbUri` input (write-only),
   2. `dbName` input.
5. Apply client validation for sink inputs:
   1. URI format check for `mongodbUri`,
   2. byte-length and character validation for `dbName`.
6. Downloadable JSON remains selectable as a structured output destination.
7. Downloadable JSON bucket/prefix is not operator-editable in pipeline UI.

### 4.2 Pipeline Detail

1. Pipeline section is editable (`name`, `mode`).
2. Search Space section is editable per 3.3 and 3.5.
3. Runtime Profile section is editable per 3.3 and 3.4.
4. Operator sink section is editable per 3.3 and 3.7.
5. IDs may be displayed as read-only metadata but are not operator-editable.
6. `source` is displayed as read-only after create.
7. `mongodbUri` is never shown in clear text after save; only presence state is shown.
8. If any run is active for the pipeline, editing controls are disabled and UI explains why.

### 4.3 Run Detail JSON Artifacts Section

1. Add `JSON Artifacts` section in Run Detail between `Execution Telemetry` and `Event Timeline`.
2. Render one artifact per row.
3. Row click opens a formatted JSON object viewer.
4. Provide row-level `Download` action for the selected artifact object.
5. Provide section-level `Download All` action for all run JSON artifacts.

### 4.4 Pipeline Delete UX

1. Delete action is available only when pipeline has no active runs and no unsettled events.
2. If active/unsettled work exists, UI must block deletion and explain required action (`cancel`, `wait for drain`).
3. Delete action requires explicit operator confirmation acknowledging full cascade purge.
4. Delete confirmation must warn that hard delete removes operational audit trail from the application.
5. UI should offer pre-delete export shortcuts:
   1. export run metadata/event index,
   2. download all JSON artifacts.

## 5) API / Contract Expectations

### 5.1 Create Pipeline

1. Client no longer sends operator-entered `searchSpaceId` and `runtimeProfileId`.
2. Control-service (or authoritative backend layer) generates these IDs.
3. If MongoDB destination is selected, client sends operator sink config with `mongodbUri` and `dbName`.
4. Generated IDs are returned in response payload as normal resource metadata.
5. `mongodbUri` is write-only and never returned in clear text by read APIs.

### 5.2 Update Pipeline Configuration

1. Update APIs must accept editable fields listed in section 3.3.
2. IDs remain immutable in update payloads.
3. Server-side validation enforces mode/output gating rules.
4. `debugLog`/`debugLogsEnabled` are not accepted from operator-facing create/update payloads.
5. Update APIs must reject attempts to mutate `source` after creation.
6. Update APIs must reject `mode` changes while a pipeline has an active run.
7. Update APIs must reject operator sink changes while a pipeline has an active run.
8. Read APIs must never return operator sink `mongodbUri` in clear text.

### 5.3 StartRun Worker Payload Routing

1. Control-service resolves operator sink for the pipeline at dispatch time.
2. Control-service must reject run start when pipeline already has an active run (`queued`, `running`, `stopping`) and return `ACTIVE_RUN_EXISTS`.
3. Before worker dispatch, control-service performs operator-sink preflight connectivity check.
4. If sink preflight fails, control-service must fail the run start and not dispatch crawler/ingestion `StartRun`.
5. Control-service injects run-scoped Mongo sink target into crawler and ingestion `StartRun` payloads.
6. Worker `StartRun` dispatch must use bounded transient retries and preserve idempotency by `runId`
   (deduplicated acceptance is valid).
7. Sink secrets must be redacted in control-service logs, error payloads, and stored run manifests.
8. `mongodbUri` must never appear in clear text outside in-memory dispatch path.
9. Worker payload contains `mongodbUri` and `dbName` for the run.
10. Worker payload uses system-defined collection names:
11. `crawl_run_summaries`
12. `ingestion_run_summaries`
13. `normalized_jobs`
14. If worker sink-client capacity guardrail is hit, run start fails fast with actionable error (`MONGODB_SINK_CAPACITY_EXCEEDED`).

### 5.4 Downloadable JSON Artifact Routing and Access

1. Control-service resolves artifact delivery from system-owned artifact config.
2. Crawler HTML dump sink and ingestion downloadable JSON sink default to GCS.
3. Worker receives artifact routing only in `StartRun`; no worker env bucket/prefix ownership.
4. Artifact path prefixes must include immutable `pipelineId` and `runId`.
5. Control-service exposes JSON artifact index and fetch/download APIs for Run Detail UI.
6. Read/download APIs return artifact payload in formatted JSON representation for single-object view.
7. Bulk download endpoint returns all run JSON artifacts as one `.zip` archive.
8. Control-service preflight validates artifact sink read/write reachability/permissions before dispatch.
9. If artifact sink preflight fails, control-service returns actionable error:
   1. `code: ARTIFACT_SINK_UNREACHABLE`
   2. `message`: human-readable reason (permission/connectivity/bucket-not-found)
   3. `details`: backend, bucket, prefix, and failing operation (`read` or `write`)
10. `download-all` must stream ZIP output (no full in-memory buffering requirement).
11. `download-all` operational limits:
    1. max archive size controlled by `CONTROL_PLANE_JSON_BUNDLE_MAX_BYTES` (default `104857600` bytes / `100MB`),
    2. max generation time controlled by `CONTROL_PLANE_JSON_BUNDLE_TIMEOUT_MS` (default `120000ms`).
12. If `download-all` exceeds limits:
    1. return `413` with `code: ARTIFACT_BUNDLE_TOO_LARGE` for size limit,
    2. return `504` with `code: ARTIFACT_BUNDLE_TIMEOUT` for timeout limit.
13. If one or more artifacts cannot be read during bundling, return `502` with
    `code: ARTIFACT_BUNDLE_PARTIAL_FAILURE` and include failed artifact IDs.
14. If run has zero JSON artifacts, return `200` with an empty ZIP archive (`application/zip`).

### 5.5 JSON Artifact API Contract Shape

1. `GET /v1/runs/:runId/json-artifacts`
   1. Query parameters:
      1. `limit` (default `50`, max `200`)
      2. `cursor`
      3. `sortBy` (`createdAt` | `fileName`)
      4. `sortDir` (`asc` | `desc`)
      5. `fileNamePrefix` (optional)
      6. default sort when omitted: `sortBy=createdAt` and `sortDir=desc`.
   1. Response shape:
      1. `runId: string`
      2. `items: Array<{ artifactId: string; fileName: string; createdAt: string; sizeBytes: number }>`
      3. `nextCursor: string | null`
2. `GET /v1/runs/:runId/json-artifacts/:artifactId`
   1. Response shape:
      1. `artifactId: string`
      2. `fileName: string`
      3. `payload: object`
3. `GET /v1/runs/:runId/json-artifacts/:artifactId/download`
   1. Returns file download for one artifact object (`application/json`).
4. `GET /v1/runs/:runId/json-artifacts/download-all`
   1. Returns `.zip` archive for all run JSON artifacts (`application/zip`).
5. Artifact metadata index is served from control-service index storage; UI must not list raw GCS objects directly.

### 5.6 Pipeline Delete API Contract

1. `DELETE /v1/pipelines/:pipelineId`
2. Preconditions:
   1. no active runs (`queued`, `running`, `stopping`),
   2. no unsettled runtime events for pipeline runs.
3. On precondition failure, return actionable error:
   1. `code: PIPELINE_DELETE_BLOCKED`,
   2. `message`: reason (active runs or unsettled events),
   3. `details`: blocking run IDs and blocking state.
4. On accepted delete request, return `202` with:
   1. `deleteJobId`
   2. `status: deleting`
5. `GET /v1/pipelines/:pipelineId/delete-status`
   1. Returns delete job status: `deleting` | `deleted` | `delete_failed`
   2. Includes progress counters and last error (if any).
6. Repeated `DELETE` while status is `deleting` returns existing `deleteJobId` and current status.
7. On success, cascade purge all pipeline-related metadata and artifacts as defined in section 3.10.
8. v2.2 deletion flow is strict coordinated rollout with no backward-compatibility window.

## 6) Validation Rules

1. Reject any request attempting to mutate system IDs.
2. Force `allowInactiveMarking = false` when destinations do not include `mongodb`.
3. For `crawl_only`, treat `ingestion.concurrency` as non-editable and non-effective.
4. For `crawl_and_ingest`, require valid `ingestion.concurrency` if ingestion execution depends on it.
5. Reject operator payloads that try to set runtime-profile debug logging flags.
6. Reject operator payloads that attempt to mutate immutable system-managed fields.
7. Reject pipeline updates while run status is `queued`, `running`, or `stopping`.
8. Reject post-create updates that attempt to change `source`.
9. When MongoDB destination is selected, require valid sink `mongodbUri`.
10. When MongoDB destination is selected, require `dbName` max `38` bytes.
11. When MongoDB destination is selected, require `dbName` charset: `A-Z`, `a-z`, `0-9`, `_`, `-`.
12. Reject operator sink updates while run status is `queued`, `running`, or `stopping`.
13. Reject responses that attempt to expose sink `mongodbUri` in clear text.
14. Reject any log/error/manifest serialization path that contains clear-text sink `mongodbUri`.
15. Reject run dispatch when operator-sink connectivity preflight fails.
16. Reject run dispatch when downloadable JSON artifact sink preflight fails.
17. Reject worker startup paths that require downloadable JSON bucket/prefix from worker env.
18. Require actionable error payload for artifact-sink preflight failures (`ARTIFACT_SINK_UNREACHABLE`).
19. Reject pipeline delete when runs are active (`queued`, `running`, `stopping`).
20. Reject pipeline delete when pipeline events are not settled.
21. Require actionable error payload for blocked deletes (`PIPELINE_DELETE_BLOCKED`).
22. Reject run start when worker sink-client capacity guardrail is exceeded (`MONGODB_SINK_CAPACITY_EXCEEDED`).
23. Validate artifact listing query params (`limit`, `cursor`, `sortBy`, `sortDir`, `fileNamePrefix`).
24. Enforce bounded sink write retry budget during active run processing:
    1. transient sink write retries are capped at default `3` retry attempts,
    2. retry backoff defaults to exponential backoff from `500ms` up to `4000ms`,
    3. on exhaustion, mark run as failed and return actionable sink-write failure.
25. Reject force-delete bypass requests in v2.2.

## 7) Migration Notes (From v2.1)

1. Remove `searchSpaceId` and `runtimeProfileId` fields from creation forms.
2. Remove `enableIngestion` from runtime profile UI and payload handling.
3. Remove runtime-profile debug logging toggle from operator UI and payload handling.
4. Introduce operator sink fields (`mongodbUri`, `dbName`) in pipeline create/edit flow.
5. Keep log verbosity configuration as service env (`LOG_LEVEL`) per service.
6. Remove worker bootstrap `MONGODB_URI` env from crawler and ingestion services.
7. Add sink-keyed shared Mongo client management in workers with acquire/release semantics.
8. Keep control-plane system sink in control-service env only and migrate to single-URI configuration.
9. Add edit controls to Pipeline Detail sections that are currently read-only.
10. Keep backward-safe rendering for existing records containing legacy fields.
11. Move downloadable JSON bucket/prefix ownership to control-service env only.
12. Remove worker-owned downloadable JSON bucket/prefix envs and rely on `StartRun.outputSinks[].delivery`.
13. Add Run Detail JSON Artifacts section with row view, single download, and download-all flows.
14. Enforce pipeline deletion precondition checks (no active runs, no unsettled events).
15. Implement cascade purge for metadata, operator sink data, HTML dumps, and JSON artifacts.
16. Release as strict coordinated rollout with no compatibility mode.
17. Introduce async delete-job status API and deletion state handling.

## 9) Acceptance Criteria

1. Operator can create a pipeline without entering any internal IDs.
2. Operator can edit Pipeline, Search Space, and Runtime Profile after creation.
3. No duplicated ingestion on/off control exists.
4. `allowInactiveMarking` cannot remain true in `downloadable_json`-only output mode.
5. Runtime-profile debug logging option is absent from operator UI and operator payload contracts.
6. UI and API validation behavior is consistent.
7. Post-create `source` edit is impossible in UI and rejected by API.
8. Pipeline configuration cannot be edited while active runs exist.
9. Operator sink can be configured by operator with `mongodbUri` and `dbName`.
10. Workers no longer depend on bootstrap `MONGODB_URI`.
11. Workers reuse one Mongo client per sink key with run-level acquire/release and idle close semantics.
12. Sink `mongodbUri` is never exposed in clear text via operator APIs/UI.
13. System sink remains control-service-owned and configured via control-service URI only.
14. Operator-sink connectivity is preflight-validated before worker dispatch.
15. `dbName` validation enforces max `38` bytes and allowed charset.
16. Sink URI is redacted in logs, errors, and manifests.
17. Downloadable JSON artifact sink is system-owned in control-service and defaults to GCS.
18. Workers receive JSON artifact routing only via `StartRun.outputSinks[].delivery`.
19. Run Detail UI exposes JSON artifacts between `Execution Telemetry` and `Event Timeline` with row view and downloads.
20. Control-service exposes JSON artifact endpoints with defined list/get/download/download-all contract shapes.
21. Dispatch-time artifact sink failures return actionable `ARTIFACT_SINK_UNREACHABLE` errors.
22. Artifact object paths include immutable `pipelineId` and `runId` prefixes for both HTML and JSON.
23. Pipeline delete is blocked unless runs are inactive and events are settled.
24. Successful pipeline delete cascades metadata, logs in system storage, operator sink data, and pipeline GCS artifacts.
25. Delete block failures return actionable `PIPELINE_DELETE_BLOCKED` errors.
26. v2.2 ships as strict coordinated rollout with no compatibility window.
27. Worker sink-client guardrail failures return actionable `MONGODB_SINK_CAPACITY_EXCEEDED` errors.
28. JSON artifact list API supports pagination, filtering, and sorting with cursor-based navigation.
29. Pipeline delete uses async job contract with observable `deleting/deleted/delete_failed` states.
30. Download-all enforces configured size/timeout limits and returns defined actionable errors.
31. Runtime sink write failures use bounded retries and escalate to actionable run failure when exhausted.

## 10) Sequence Flows

### 10.1 Start Run (control-service orchestrated)

1. Operator triggers `Start Run` from Control Center.
2. Control-service loads pipeline config and verifies:
   1. pipeline is not in `deleting` state,
   2. pipeline has no active run (`queued`, `running`, `stopping`).
3. Control-service validates pipeline mode and editable constraints from section 6.
4. Control-service resolves sinks for this run:
   1. operator Mongo sink (if configured),
   2. system artifact sink (GCS default),
   3. run-scoped prefixes including immutable `pipelineId` and `runId`.
5. Control-service runs preflights:
   1. worker availability check,
   2. operator sink connectivity check (if Mongo output selected),
   3. artifact sink reachability/permissions check.
6. If any preflight fails, return actionable error and do not dispatch workers.
7. If mode is `crawl_and_ingest`, dispatch ingestion `StartRun` first.
8. Ingestion `StartRun` dispatch uses transient retry policy and idempotent `runId` semantics
   (`deduplicated` response state is valid).
9. After ingestion accepts/deduplicates (or if mode is `crawl_only`), dispatch crawler `StartRun`.
10. Crawler emits events and artifacts under run-scoped routes.
11. If crawler `StartRun` fails after ingestion accepted, control-service triggers ingestion cancel with
    `reason: startup_rollback`.
12. If startup rollback cancel fails after retries, control-service marks run failed with
    `startup_rollback_cancel_failed`.
13. Ingestion processes run events and writes outputs to resolved operator sink and artifact sink routes.
14. Run finalization updates run projection and summaries; sink clients are released.

### 10.2 Pipeline Delete (async cascade)

1. Operator requests `DELETE /v1/pipelines/:pipelineId`.
2. Control-service checks preconditions:
   1. no active runs (`queued`, `running`, `stopping`),
   2. no unsettled runtime events.
3. If blocked, return `PIPELINE_DELETE_BLOCKED` with blocking details.
4. If eligible, create delete job and return `202` with `deleteJobId` and `status: deleting`.
5. Delete job executes cascade purge:
   1. control-plane metadata (pipeline, runs, event projections, stored logs),
   2. operator sink pipeline data,
   3. GCS artifacts under pipeline prefix (HTML + downloadable JSON).
6. On transient failure, retry with bounded exponential backoff.
7. Job ends in `deleted` or `delete_failed`; status is exposed via delete-status endpoint.

### 10.3 Download All JSON Artifacts

1. UI calls `GET /v1/runs/:runId/json-artifacts/download-all`.
2. Control-service loads artifact index for the run.
3. If artifact index is empty, return `200` with an empty ZIP (`application/zip`).
4. Otherwise, control-service streams artifacts into ZIP output (`application/zip`).
5. Enforce size and timeout limits during streaming.
6. If limits are exceeded, return defined actionable errors.
7. If some artifacts fail to read, return `ARTIFACT_BUNDLE_PARTIAL_FAILURE` with failed IDs.

## 11) Data Models

```ts
export type PipelineMode = 'crawl_only' | 'crawl_and_ingest';

export type DeleteJobState = 'deleting' | 'deleted' | 'delete_failed';

export type ApiErrorCode =
  | 'ARTIFACT_SINK_UNREACHABLE'
  | 'ARTIFACT_BUNDLE_TOO_LARGE'
  | 'ARTIFACT_BUNDLE_TIMEOUT'
  | 'ARTIFACT_BUNDLE_PARTIAL_FAILURE'
  | 'PIPELINE_DELETE_BLOCKED'
  | 'MONGODB_SINK_CAPACITY_EXCEEDED';

export interface OperatorMongoSinkWriteModel {
```

## 12) v2.2 Non-goals

1. Secret vault integration, encryption-at-rest design, and rotation workflows for operator `mongodbUri`.
2. Contract versioning/backward-compat windows for `StartRun` payloads (v2.2 uses strict coordinated rollout).
3. Pub/Sub architecture redesign, queue buffering strategies, or backlog persistence changes.
4. Operator-request run cancellation semantics (beyond currently defined cancel-and-drain behavior).
5. IAM/lifecycle-policy hardening for GCS buckets and advanced multi-tenant isolation.
6. Advanced artifact index reconciliation jobs between index storage and raw GCS listing.
7. Output backend expansion beyond current MongoDB + downloadable JSON options.
